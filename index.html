第 1 章 · 选型决策

Claude Chat 还是 Claude Cowork？

Claude Chat

"当下"向 Claude 提问、和它一起工作
  • 一次性、装得进单次对话的任务
  • 提问 / 检索摘要 / 就地起草一段文字
  • 每次对话独立，不适合长期跟进的事项
  • 新用户的第一站
VS

Claude Cowork

把一个项目"派"给 Claude，事后审阅
  • 横跨多文件、多应用、多步骤的事项
  • 审阅整个数据室合同 / playbook 比对
  • 可跑数分钟到数小时，由律师审阅产出
  • 接 iManage / NetDocuments / Box / M365
同一个工具，上午帮你修订 NDA，下午就能清空收件箱、为明天的取证做准备——Cowork 尤其擅长这种多应用、多步骤的"工作开销"。

第 1 章 · 进阶机制

子代理与插件：如何安全地扩展能力

子代理 Subagents

如果说 Skill 告诉 Claude "怎么做"，子代理就是在独立上下文窗口、用自己的系统提示词与工具权限运行的"职责狭窄助手"。

完成一个有边界的任务（核对引证、抽取条款、审核术语）后回报。既能避免长事项把单一上下文压爆，也能在敏感系统环节施加更严格的工具限制。

插件即"执业领域整包"

插件把以下三类能力捆成一个可安装包，让一个执业领域开箱即带上"该有的好帮手"：

Skill · 怎么做
子代理 · 委派助手
连接器 · 接数据
审批流 · fork 自定义

合作伙伴插件则把专有数据、供应商定制 Skill 引入 Claude。

三阶段推广节奏：先打地基，再聚焦试点，最后规模化

1

打地基

Foundation
  • 铺设访问与连接器：API / Bedrock / Vertex / Foundry
  • 配齐 SSO、SCIM、审计日志、自定义保留
  • 治理并行：法律保留、特权保护、隐私审查
  • 用 Claude 分析工单，破解"冷启动"
2

试点

Pilot
  • 先锋用真实事项数据跑真实工作流
  • 按既定标准衡量：时间节省 + 采用率
  • 上线顺序：Skill/插件 → M365 加载项 → Cowork
  • 每周短会，及时捕获引证幻觉、辖区差异
3

规模化

Scale
  • 经管理员市场把插件铺给更多团队
  • 新员工入职第一天即用编码好的工作流
  • Skill 跨执业领域复用，产生复利
  • 就治理框架达成全所共识

第 4 章 · 安全合规

面向 CIO 与 IT 负责人的常见问答

Q我们的数据会被拿去训练模型吗？
不会。Anthropic 不用 Claude.ai / Cowork 上 Enterprise Plan 账户的输入或输出做训练。
Q支持零数据保留（ZDR）吗？
Platform（API）与 Code 对已审批客户可用；Claude.ai / Cowork 为有状态产品，支持自定义保留窗口，可短到 30 天。
Q律师-客户特权如何保护？
连接器尊重 DMS / 事项系统已配置的访问控制，不用客户数据训练，企业版支持自定义保留；律所通常再叠加自家处理政策。
Q支持哪些身份与访问控制？
SAML SSO、SCIM 用户预置、RBAC 角色化访问，以及管理员管控的插件市场。
QCowork 如何访问本地文件？
只读取用户显式授权的文件夹，按用户范围控制，不在后台索引硬盘。
Q部署在哪里？要装什么？
Anthropic 托管 SaaS；需留云边界内则经 Bedrock/Vertex/Foundry 自建。Cowork 为签名桌面应用，可经 MDM 管理。
